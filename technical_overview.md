# Технический обзор архитектуры YaRag

В данном документе описан полный цикл работы системы: от этапа автономной оптимизации (обучения) до обработки пользовательского запроса в реальном времени. Архитектура построена на принципах **Modular RAG** и вдохновлена пайплайном сервиса Яндекс Нейро.

---

## Часть 1: Оптимизация системы (Offline / Optimization Phase)

В отличие от классических систем, где промпты пишутся вручную, YaRag использует **программную оптимизацию**.

### 1. Подготовка данных (Gold Dataset)
Используется файл `gold_data.json`, содержащий эталонные пары "Запрос — Ожидаемые источники (URL) — Эталонный ответ". Это "учебник", по которому система тюнингует сама себя.

### 2. Оптимизация инструкций (DSPy + MIPROv2)
Вместо того чтобы мы сами подбирали слова в системном промпте, мы используем фреймворк **DSPy**:
*   **MIPROv2 (Multi-objective In-Context Prompt Optimization)**: Алгоритм генерирует десятки вариаций инструкций для LLM (генератора и судьи).
*   Система прогоняет эти промпты через `gold_data`, замеряя метрики (Overlap, Recall).
*   Выбирается та комбинация инструкций, которая дает максимальное совпадение с эталоном. Результат сохраняется в `config/prompts_prod.json`.

### 3. Подбор гиперпараметров (Optuna)
Размер текстового фрагмента (**Chunk Size**) и их перекрытие (**Overlap**) критически влияют на то, сколько полезной информации попадет в LLM.
*   **Optuna** запускает сотни испытаний (trials), варьируя эти параметры.
*   Цель: найти такой размер чанка, при котором семантический поиск (Ranker) наиболее точно находит нужные куски текста.
*   Оптимальные значения (например, 450 знаков с перекрытием 40) сохраняются в `config/rag_config.json`.

## Часть 2: Цикл обработки запроса (Real-time Pipeline)

Когда пользователь вводит запрос, система проходит через 8 технологических слоев:

### 1. Восстановление контекста (Query Rephrasing)
LLM анализирует историю диалога. Если пользователь спрашивает "А как там погода?", система превращает это в полноценный запрос "Погода в Екатеринбурге на 27 марта 2026", чтобы поисковик выдал релевантные ссылки.

### 2. Мульти-запросный поиск (Search Expansion)
Система генерирует 2-3 вариации поискового запроса для расширения охвата. Запросы уходят в **XMLRiver** (который проксирует выдачу Яндекса). Мы получаем топ-10/15 ссылок с их сниппетами.

### 3. Интеллектуальный скрапинг (Smart Scraper & Anti-Bot)
Самый сложный этап. Обычные скрипты часто блокируются (403/429 ошибки).
*   **Scrapfly API**: Используется для контента, защищенного Cloudflare.
*   **Параллельная загрузка**: Все страницы грузятся одновременно с таймаутом 30 сек на страницу.
*   **Fallback**: Если страница не отдала контент (таймаут или блок), система не падает, а использует сниппет из поиска как резервный источник.

### 4. Сегментация и Чанкирование (Sliding Window)
Текст страниц режется на чанки по параметрам, найденным на этапе оптимизации. Используется метод "скользящего окна", чтобы смысл предложений не терялся на границах кусков.

### 5. Гибридное ранжирование (Hybrid Search)
Для каждого чанка считается интегральный скор:
1.  **Semantic Score (Yandex Embeddings)**: Насколько кусок текста близок к вопросу по смыслу (косинусное сходство векторов).
2.  **Keyword Score (BM25)**: Совпадение по точным словам (важно для дат, имен, моделей).
3.  **Domain Boosting**: Авторитетные сайты (Wikipedia, РБК и др.) получают бонусные баллы.
4.  **Diversity Penalty**: Штраф за слишком большое количество чанков с одного и того же сайта.

### 6. Отбор победителей (Winner Selection)
LLM в роли "судьи" (Judge) просматривает топ-20 кандидатов и выбирает **5 финальных источников**, которые лучше всего дополняют друг друга и содержат минимум "шума".

### 7. Конденсация (SelectGlue)
Для каждого из 5 победителей мы не берем всю страницу (она может быть огромной). Система выбирает только самые "сочные" куски и склеивает их в компактное саммари для LLM.

### 8. Генерация ответа с цитированием (Grounded Generation)
Финальная стадия. YandexGPT получает 5 сжатых источников и строгие инструкции:
*   Отвечать **только** на основе предоставленного текста.
*   Обязательно ставить ссылки на источники в формате `[1]`, `[2]`.
*   Если информации нет — честно об этом сказать.

## Часть 3: Инфраструктурная обвязка

*   **Redis**: Кэширует результаты поиска и ответы LLM на 24 часа. Это снижает нагрузку на API и ускоряет повторные запросы почти мгновенно.
*   **Docker Compose**: Весь стек (Бэкенд на Python/FastAPI, Фронтенд на React/Nginx, Redis) разворачивается одной командой, работая в изолированной сети.
*   **Security Layers**: Никаких захардкоженных ключей в коде — всё передается через переменные окружения и контролируемую среду `.env`.

Это превращает YaRag из простого скрипта в надежную систему, которая сама обучается на данных и умеет добывать информацию даже с труднодоступных ресурсов.
